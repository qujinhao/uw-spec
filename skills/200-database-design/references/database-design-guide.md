# 数据库设计指南

本指南定义数据库设计的规则、规范和最佳实践。
设计使用的数据库类型为 MySQL 8.4+，数据库编码为 UTF8MB4。
---

## 1. 命名规范

### 1.1 核心规则

**表名格式**: `{module}_{实体}` / `{module}_{主实体}_{从实体}` / `{module}_{实体1}_{实体2}_ref` — 三种模式，全小写，2-4段。

- **模块**：单个单词，全小写
- **实体**：单个单词，全小写（如果实在无法避免多单词，不得超过2个单词，使用snake_case命名）
- **主实体**：父级/主实体，用于表达从属关系
- **从实体**：子级/从实体，表达从属于主实体

| 表类型 | 格式 | 示例 | 说明 |
|--------|------|------|------|
| 模块主表 | `{module}_{实体}` | `guest_info`, `product_info` | 模块最核心的业务实体；统一使用 `info` 作为实体名，避免 `product_product` / `guest_guest` 等模块名与实体名重复的命名 |
| 独立实体表 | `{module}_{实体}` |  `product_category`,`guest_login_log` | 业务核心实体，无父级从属 |
| 主从表 | `{module}_{主实体}_{从实体}` | `mall_order_item`, `mall_cart_item` | 从属于主表的明细数据 |
| 关联表 | `{module}_{实体1}_{实体2}_ref` | `guest_user_role_ref` | 多对多关系映射表 |


**其他命名对象**：

| 对象 | 格式 | 示例 |
|------|------|------|
| 主键 | `id` | `id` |
| 字段名 | `snake_case` | `guest_id`, `order_state`, `create_date` |
| 外键字段 | {entity}_id | `guest_id`, `order_id`, `product_id` |
| 普通索引 | `idx_{表名缩写}_{字段名}` | `idx_gu_mobile` |
| 唯一索引 | `uk_{表名缩写}_{字段名}` | `uk_gu_email` |

### 1.3 字段命名规范

#### 通用字段（所有表必须包含）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | BIGINT | 主键，自增 |
| `saas_id` | BIGINT | 租户ID（SaaS必加） |
| `create_date` | DATETIME(3) | 创建时间 |
| `modify_date` | DATETIME(3) | 修改时间 |
| `state` | INT | 逻辑状态（0禁用/1启用） |


#### 字段命名规则

| 规则 | 示例 | 说明 |
|------|------|------|
| 布尔字段 `is_` 前缀 | `is_active`, `is_paid` | INT，0=否 1=是 |
| 时间字段 `_date` 后缀 | `create_date`, `pay_date` | DATETIME(3) |
| 价格字段 `_price` 后缀 | `product_price` | BIGINT（分） |
| 金额字段 `_amount` 后缀 | `order_amount` | BIGINT（分） |
| 数量字段 `_count` 后缀 | `view_count` | INT |
| 外键字段 `_id` 后缀 | `guest_id` | BIGINT |
| 状态字段 `state` | `order_state` | INT |
| IP 字段 `_ip` 后缀 | `user_ip` | VARCHAR(50)，IPV4/V6地址 |
| 类型字段 `_type` 后缀 | `pay_type` | INT |
| 编码字段 `_no` 后缀 | `order_no` | VARCHAR，业务唯一编码 |
| 名称字段 `_name` 后缀 | `product_name` | VARCHAR(100) |
| 描述字段 `_desc` 后缀 | `product_desc` | TEXT |
| 标题字段 `_title` 后缀 | `article_title` | VARCHAR(200) |
| 内容字段 `_content` 后缀 | `msg_content` | TEXT |
| 图片字段 `_icon` / `_img` / `_imgs` 后缀 | `user_icon` | VARCHAR(500) |
| 公告字段 `notice_` 前缀| `notice_title` | VARCHAR(200) | 
| 审核字段 `audit_` 前缀 | `audit_state`, `audit_user_id`, `audit_user_ip`, `audit_user_info`, `audit_remark`, `audit_date` | INT，注释注明枚举值 |
| 关联/引用字段 `ref_` 前缀 | `ref_type` varchar(50), `ref_id` BIGINT | 关联ID根据情况，，可以用VARCHAR(20)或BIGINT |
| 供应商字段 `supp_` 前缀 | `supp_id` | 供应商字段都使用supp_开头 |
| 分销商字段 `dist_` 前缀 | `dist_id` | 分销商字段都使用dist_开头 |
| 排序字段 `sort_rank` | `sort_rank` | bigint 排序字段 |
| 备注字段 `remark` | `remark` | VARCHAR(500) |
| 扩展字段 `ext_data` | `ext_data` | JSON |
| 多语言字段 `lang` | `lang` | VARCHAR(10)，如 zh-CN, en-US, ja-JP |
| 多语言翻译表 | `{主表名}_lang` | `product_info_lang` | 主表需要翻译的字段独立存储，1:N 关系 |

### 1.4 索引命名规范

**索引名格式**:
- 普通索引：`idx_{表名缩写}_{字段名}`
- 唯一索引：`uk_{表名缩写}_{字段名}`
- 组合索引：`idx_{表名缩写}_{字段1}_{字段2}`

**表名缩写规则**：取各单词首字母，无下划线
- guest_user → gu
- order_info → oi
- order_item → oitem（避免与oi冲突）
- shopping_cart_item → sci

## 2. 设计流程

### 2.1 从 PRD 提取业务实体

| PRD 位置 | 提取内容 | 示例 |
|----------|---------|------|
| README.md 包含模块 | 各终端模块清单 | 用户中心 → 用户实体 |
| domains/M{seq}*.md 功能列表 | 领域模块核心数据对象 | 订单管理 → 订单实体 |
| M{seq}P{seq}*.md 页面元素 | 页面展示/操作的数据 | 商品列表 → 商品实体 |
| M{seq}P{seq}*.md 搜索/筛选 | 查询条件 → 索引依据 | 状态筛选 → state 索引 |

### 2.2 覆盖度验证

每个 PRD 功能必须对应至少一个实体，覆盖率 = 100%

| 模块 | 功能 | 依赖实体 | 是否覆盖 |
|------|------|---------|---------|
| 用户中心 | 注册/登录 | guest_user | ✅ |

### 2.3 E-R 关系定义

使用标准符号描述实体关系：
- `||--o{` : 1对多
- `}o--||` : 多对1
- `}o--o{` : 多对多

## 4. 索引设计规则

### 4.1 必须创建索引的场景

- 主键（自动创建）
- 外键字段
- WHERE 条件高频字段
- ORDER BY 字段
- 业务唯一字段

### 4.2 禁止创建索引的场景

- 低基数字段（如性别、布尔值）
- 频繁更新的字段（谨慎）
- 长文本字段（使用前缀索引）

### 4.3 组合索引设计原则

- 遵循最左前缀原则
- 等值查询字段放前面
- 范围查询字段放后面
- 区分度高的字段放前面

### 4.4 索引优化规则

| 规则 | 说明 |
|------|------|
| 覆盖索引 | 查询字段都在索引中，避免回表 |
| 避免函数操作 | `WHERE DATE(create_date) = 'xxx'` 会导致索引失效 |
| 避免隐式转换 | 字段类型与查询值类型必须一致 |
| 控制索引数量 | 单表索引不超过 5-7 个 |

## 5. 非功能设计规范

### 5.1 数据规模评估标准

| 数据量级 | 设计策略 |
|---------|---------|
| < 500万 | 单表 + 索引优化 |
| 500万 - 5000万 | 分区或归档 |
| > 5000万 | 分库分表 |

### 5.2 分库分表策略

| 策略 | 适用场景 | 分片键 |
|------|---------|--------|
| 按租户分库 | SaaS多租户 | saas_id |
| 按时间分表 | 日志、订单 | create_date（按月） |
| 按ID取模 | 用户表 | user_id |
| 按地区分库 | 地域性业务 | region_code |

### 5.3 数据归档规则

| 归档类型 | 触发条件 | 实现方式 |
|---------|---------|---------|
| 定期归档 | 数据超过保留期 | 定时任务迁移到归档表 |
| 冷热分离 | 3个月以上数据 | 热库+冷库分离 |
| 逻辑删除清理 | state=0 且超过保留期 | 物理删除或归档 |

### 5.4 安全规范

| 安全措施 | 实现要求 |
|---------|---------|
| 敏感字段加密 | 手机号、身份证 AES 加密 |
| 密码存储 | BCrypt 哈希 + 盐 |
| 数据脱敏 | 日志中敏感信息脱敏 |
| 访问控制 | 最小权限原则 |

### 5.5 高可用要求

- 主从复制架构
- 全量备份：每周一次
- 增量备份：每日一次
- Binlog 备份：实时

## 6. 设计原则

| 原则 | 说明 |
|------|------|
| 第一范式 | 字段原子性，不可再分 |
| 第二范式 | 消除部分依赖 |
| 第三范式 | 消除传递依赖 |
| 反范式化 | 适当冗余减少关联查询（谨慎使用） |

## 7. 约束规范

- **禁止使用外键约束**，由程序控制外键关系
- 所有表必须包含 `saas_id` 字段（SaaS架构）
- 所有表必须包含 `create_date`, `modify_date`, `state` 通用字段
- 金额/价格使用 BIGINT 类型，单位为分
- 状态字段使用 INT，注释中说明枚举值含义
- 每个字段必须有 COMMENT 注释
- 表注释格式为"模块-实体"

## 8. 多语言翻译表设计规范

### 8.1 适用场景

当主表存在需要多语言翻译的文本字段（如名称、描述、标题等）时，创建对应的 `_lang` 翻译表。

### 8.2 命名规则

| 对象 | 格式 | 示例 |
|------|------|------|
| 翻译表名 | `{主表名}_lang` | `product_info_lang` |
| 外键字段 | `{主实体}_id` | `product_id`（关联 `product_info.id`） |
| 语言字段 | `lang` | `VARCHAR(10)`，如 `zh-CN`、`en-US` |

### 8.3 结构约束

| 约束 | 说明 |
|------|------|
| 翻译表只存需要翻译的字段 | 价格、数量等非文本字段不出现 |
| 主表必须有 `lang` 字段 | 记录该条数据的默认语言 |
| 翻译表必须有 `lang` 字段 | 记录翻译目标语言 |
| 外键字段使用 `{主实体}_id` | 如 `product_id`、`order_id`，不用 `ref_id` |
| UNIQUE KEY (`{主实体}_id`, `lang`) | 每条主数据每种语言只有一条翻译 |
| 无外键约束 | 由程序控制关联关系 |

### 8.4 DDL 模板

参见 [database-ddl-template.md](database-ddl-template.md) §2.3 多语言翻译表模板。

### 8.5 查询策略

使用 `LEFT JOIN + COALESCE` 实现自动降级：

| 设计决策 | 说明 |
|---------|------|
| JOIN 方式 | LEFT JOIN（保留无翻译的主表数据） |
| 降级函数 | `COALESCE(l.field, t.field)`，翻译为空时回退主表默认值 |
| JOIN 条件 | `l.{entity}_id = t.id AND l.lang = ? AND l.state = 1` |
