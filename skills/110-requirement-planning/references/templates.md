# 需求规划模板

## 目录结构

```
requirement/
├── prds/                              # PRD产品文档
│   ├── README.md                      # 项目总览（唯一汇总文档）
│   ├── domains/                       # 业务领域（整合多端共享的业务对象和规则）
│   │   ├── M01-{module}.md            # 领域模块文档
│   │   └── ...
│   ├── {role}-{platform}/                  # 前端项目
│   │   ├── M01-{一级菜单名}/           # M=一级菜单/功能组
│   │   │   ├── M01P01-{页面组名}.md    # P=二级菜单/页面组（含多个视图）
│   │   │   ├── M01P02-{页面组名}.md
│   │   │   └── ...
│   │   ├── M02-{一级菜单名}/
│   │   │   └── ...
│   │   └── ...
│   └── reviews/                       # 评审报告
│       └── REVIEW-PRD-YYMMDDHHMM.md
└── interviews/                        # 访谈记录
    └── interview-YYMMDDHHMM.md
```

**核心概念**：

| 概念 | 含义 | 导航对应 | 文件结构 |
|------|------|---------|---------|
| 领域M | 领域模块，整合多端同类需求 | 无（后端概念） | `domains/M{seq}-{name}.md` |
| 前端M | 一级菜单/功能组 | 侧边栏菜单/底部Tab | `M{seq}-{name}/` 文件夹 |
| 前端P | 二级菜单/页面组（一组相关视图） | 二级菜单或独立功能页 | `M{seq}P{seq}-{name}.md` |

**前端M与领域M的关系**：两套独立编号，通过映射表关联。多个前端M可能对应同一个领域M。

**命名规则**：
| 文件 | 命名 | 示例 |
|------|------|------|
| 领域模块 | `M{seq}-{中文模块名}.md` | `M01-用户与认证.md` |
| 前端M文件夹 | `M{seq}-{中文菜单名}/` | `M03-POI管理/` |
| 前端P文档 | `M{seq}P{seq}-{中文页面名}.md` | `M03P01-POI管理.md` |

### 交互流程

> 按领域模块组织，每个流程用 Mermaid 时序图描述端与领域模块之间的交互。participant 必须使用具体模块名（如 `M01-用户与认证`），禁止使用模糊的"业务领域"。

**流程图设计规范**：

| 规则 | 说明 |
|------|------|
| 核心领域必须有流程图 | 涉及用户核心价值链的领域（如行程管理、订单支付）必须有交互流程图，无论是否跨端 |
| 跨端联动必须有流程图 | 多端之间存在数据同步、状态联动、审批流转等行为时必须有流程图 |
| 单端核心流程也要覆盖 | 即使只有单端参与，只要涉及多步骤状态流转（如行程的创建→预订→执行→完成），也需要流程图 |
| 流程图需完整覆盖业务链路 | 流程图应覆盖从触发到完成的完整业务链路，不截断。如行程流程应包含创建→预订→执行→分享 |
| participant 使用具体模块名 | 时序图中 participant 必须写具体模块名（如 `M02-行程管理`），涉及跨模块协作时列出所有相关模块 |

**{流程名称}**（{对应领域M}）：

​```mermaid
sequenceDiagram
    actor 用户 as guest用户
    participant Guest as guest端
    participant M{seq} as M{seq}-{module}
    participant SaaS as saas端
    actor 管理员 as saas管理员

    用户->>Guest: {操作}
    Guest->>M{seq}: {请求}
    M{seq}-->>SaaS: {联动}
    SaaS-->>管理员: {展示}
​```
