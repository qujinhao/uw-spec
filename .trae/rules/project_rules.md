# 项目规则 - Skill 编写规范

## Skill 文件结构规范

### 目录结构
```
skill-name/
├── SKILL.md              # 主入口文件（必须）
├── references/           # 详细文档（可选）
│   ├── template.md       # 输出模板
│   ├── examples.md       # 示例代码
│   └── api.md            # API 文档
└── scripts/              # 辅助脚本（可选）
    └── helper.py
```

### SKILL.md 头部格式
```yaml
---
name: skill-name
description: 技能功能 + 使用场景 + 具体触发条件（数字编号）
---
```

## 内容规范

### 1. Description 规范
- **长度**：≤ 300 字符
- **格式**：技能功能 + 使用场景 + 具体触发条件
- **触发条件**：使用数字编号 (1)(2)(3)

**示例**：
```
综合PDF编辑、转换和文本提取技能。当Claude需要处理PDF文件时使用：
(1)编辑内容，(2)转换格式，(3)提取文本，或其他PDF任务
```

### 2. 文件大小限制
| 指标 | 限制 |
|------|------|
| 总行数 | ≤ 600 行（理想 <=300 行） |
| 总词数 | < 5000 词 |
| 代码块 | 精简，长代码移到 references/ |

### 3. 内容分层原则

| 保留在 SKILL.md | 移到 references/ |
|----------------|-----------------|
| 核心工作流程 | 详细示例代码 |
| 触发条件 | 配置选项详解 |
| 选择指导 | API 文档 |
| 资源引用链接 | 变体细节 |

## 编写风格

### 1. 使用表格替代段落
- 配置项 → 表格
- 参数说明 → 表格
- 角色职责 → 表格
- 技术栈 → 表格

### 2. 弱指令转强指令
| ❌ 弱指令 | ✅ 强指令 |
|----------|----------|
| "尽量保持函数小" | "函数 >50 行需添加合理性注释" |
| "使用好的命名规范" | "布尔变量：is/has/should 前缀" |
| "注意性能" | "批量操作单次 ≤1000 条" |

### 3. 删除冗余内容
**禁止包含**：
- 通用 AI 能力（"写干净代码"）
- 普遍最佳实践（DRY、SOLID、KISS）
- 自明的代码信息（使用的编程语言）
- 父级已定义的规则
- **出现纯数字的技能名**,如210，310等。

## 质量检查清单

创建或修改 skill 后，检查以下项目：

- [ ] description ≤ 300 字符
- [ ] description 包含1-3个具体触发条件
- [ ] SKILL.md ≤ 600 行
- [ ] 使用表格替代长段落
- [ ] 指令具体可执行（无量化词）
- [ ] 无通用 AI 能力描述
- [ ] 无普遍最佳实践说明
- [ ] 文件引用路径正确

## REVIEW 循环（必须执行）
每次任务完成后，必须进行三次 REVIEW 循环，直到任务质量达标。
- 第一次：主开发完成后的内部审核，确认技能文件合理，没有重复逻辑，多技能之间保持一致性。
- 第二次：站在资深架构师的角度，确认任务是否符合需求。
- 第三次：站在资深CTO的角度，确认任务是否符合业务需求。
