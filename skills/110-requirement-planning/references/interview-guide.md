## 访谈原则

| 原则 | 执行方式 |
|------|---------|
| 不假设，只验证 | 每个需求点都需用户确认，从不凭空推测 |
| 逐轮推进 | 每轮聚焦1个层次，避免一次性抛出所有问题 |
| 确认理解 | 每轮结束前用"让我重复一遍我理解的需求..."复述 |
| 启发式追问 | 使用苏格拉底式提问激发用户思考，而非机械问答 |
| 记录决策 | 每轮将确认信息整理为结构化笔记 |
| 用户确认 | 用户明确说"确认"或"没问题"后才进入下一轮 |

## 1.1 项目背景访谈（Discovery）

目标：理解业务上下文、成功标准和约束条件。

### 执行方式（重要）

**逐个提问，每问一题等待回答**：

| 步骤 | 操作 |
|------|------|
| 1 | 使用 `AskUserQuestion` 工具呈现第1个问题（可点击选择框） |
| 2 | **等待用户选择或自由录入** |
| 3 | 根据回答进行必要的启发式追问 |
| 4 | 确认理解正确 |
| 5 | 使用 `AskUserQuestion` 呈现下一个问题 |
| 6 | 重复直到6个问题全部完成 |

**禁止行为**：
- 一次性列出Q1-Q6让用户逐一回答
- 纯文本列出选项让用户手工录入（必须用选择框）

### 必问清单

每个问题使用 `AskUserQuestion` 呈现选择框。选项为通用参考，根据上下文动态调整。

| # | header | question | 参考选项（可动态调整） | multiSelect |
|---|--------|----------|----------------------|-------------|
| 1 | 触发因素 | 为什么现在要做这个产品？ | 现有方案效率低/体验差、开拓新市场/新用户、内部管理需求、竞品压力 | true |
| 2 | 成功指标 | 6个月后如何衡量成功？ | 用户/客户数量、收入/订单量、用户满意度、运营效率提升 | true |
| 3 | 目标用户 | 谁会使用这个产品？ | 企业内部员工、企业客户/用户、渠道商/合作伙伴、普通消费者 | true |
| 4 | 现有方案 | 目前用户如何解决这个问题？ | 人工处理/Excel、使用其他软件/系统、微信/群沟通、暂无方案 | true |
| 5 | 时间要求 | 有没有明确的上线时间要求？ | 有明确deadline（请说明）、尽量快、暂无要求、视情况调整 | false |
| 6 | 合规要求 | 是否涉及合规要求？ | 无特殊要求、数据隐私、支付合规、行业监管要求 | true |

**追问触发条件**：

| 用户回答 | 追问方向 |
|---------|---------|
| 选择多个选项 | "哪项是最核心的？" |
| 选择"Other"自由录入 | "能具体描述一下吗？" |
| 回答笼统 | "能举一个具体场景吗？" |
| 选择"暂无方案" | "那平时是怎么处理的？" |

### 常见误区提醒

| 误区 | 正确做法 |
|------|---------|
| 用户说"做一个类似XX的系统" | 追问"跟XX相比，您希望哪些方面做得不同或更好？" |
| 用户描述的是解决方案而非问题 | 追问"这个功能解决的是什么场景下的什么问题？" |
| 目标用户过于宽泛 | 追问"第一批使用的用户会是哪些人？" |
| 用户直接跳到功能细节 | 温和引导"在聊具体功能之前，能先说说为什么要做这个吗？" |

## 1.2 需求探索与头脑风暴（Brainstorming）

目标：在理解项目背景后、锁定具体需求前，主动探索产品可能性，发现用户未提及的需求。

### 前置条件

阶段1.1（Discovery）已完成，用户已确认项目背景和核心痛点。

### 执行方式（重要）

**逐个环节执行，每完成一个环节确认后再进入下一个**：

| 步骤 | 操作 |
|------|------|
| 1 | 使用 `AskUserQuestion` 呈现环节一的选择题 |
| 2 | **等待用户选择或自由录入** |
| 3 | 用户确认后，进入环节二 |
| 4 | 重复直到所有环节完成 |

**禁止行为**：
- 一次性列出所有环节的所有问题让用户回答
- 纯文本列出选项让用户手工录入（必须用选择框）

### 环节一：用户场景扩展

列举5-10个典型和边缘用户场景，发现用户未提及的需求。

**执行步骤（逐个确认）：**

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 基于阶段1.1信息，先提出1-2个典型场景 | "比如，用户早上打开APP会做什么？" |
| 2 | **等待用户确认或补充** | 根据反馈调整 |
| 3 | 继续追加其他典型场景 | 逐个展示，逐个确认 |
| 4 | 追加边缘场景 | 提出1个，确认后再提下一个 |
| 5 | 最后询问 | "还有遗漏的场景吗？" |

**话术示例：**

向用户展示梳理的场景时，使用以下结构：

**开场**：基于我们之前聊到的信息，我梳理了以下用户场景：

**典型场景**：
1. 销售代表早上打开APP，查看今日待跟进客户列表
2. 销售代表完成一次客户拜访后，在手机上录入跟进记录
3. 销售经理查看团队本周的跟进完成率

**边缘场景**：
4. 销售代表出差时网络不稳定，需要离线记录
5. 客户信息涉及隐私，经理只能看到脱敏数据

**确认问题**：还有遗漏的场景吗？

### 环节二：竞品功能映射

分析同类产品功能清单，标注行业标配和差异化机会。

> **跳过条件**：如果用户表示没有可参照的竞品（企业内部系统、创新性产品等），标记为"无竞品参考"并跳过本环节，直接进入环节三。

**执行步骤：**

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 确认竞品范围 | 询问用户"市面上有哪些产品在做类似的事情？" |
| 2 | 列出竞品核心功能 | 按功能域分组列出，标注哪些是行业标配 |
| 3 | 差异化分析 | 标注竞品做得不好的点，与用户痛点交叉验证 |
| 4 | 确认采纳建议 | 将竞品功能分为"必须对标"、"可选借鉴"、"不做"三类 |

**竞品功能对比表格式：**

向用户展示竞品分析结果时，使用以下结构：

**标题**：竞品功能对比

**表格**：
- 表头：功能域 | 行业标配 | 竞品A | 竞品B | 我们的差异点
- 行内容：按功能域填充实际对比内容

**示例行**：
- 订单管理 | 订单列表、详情、状态跟踪 | ✓ | ✓ | -
- 数据报表 | 基础统计 | 仅有图表 | 支持导出 | 支持AI分析建议

**建议分类**：
- ✅ 必须对标（行业标配）：订单管理、用户管理
- ⭐ 差异化机会：AI分析、自动化提醒
- 📋 可选借鉴：导出功能、自定义看板

### 环节三：SCAMPER 发散

用结构化方法打破思维定式，生成3-8个创新功能创意。

**SCAMPER 方法：**

| 维度 | 提问 | 示例（电商场景） |
|------|------|-----------------|
| S - 替代 | "能否用X替代Y？" | 用AI推荐替代手动分类浏览 |
| C - 组合 | "能否将A和B组合？" | 购物车+社交分享 = 拼团 |
| A - 借用 | "其他行业是怎么做的？" | 借用游戏的签到奖励机制 |
| M - 修改 | "修改某属性的值？" | 把按月订阅改为按使用量付费 |
| P - 其他用途 | "现有功能还能用在哪？" | 物流追踪用于预估到达时间推荐 |
| E - 消除 | "去掉什么反而更好？" | 去掉注册步骤，手机号一键登录 |
| R - 反转 | "反过来会怎样？" | 用户定价、商家竞标 |

**执行方式：** 从上表中选取 2-3 个最相关的维度，用对应的提问模板向用户提问，收集创意后请用户筛选值得深入的方向。

### 环节四：技术赋能探索

思考AI、实时、自动化等技术能为产品带来什么创新。

**探索维度：**

| 技术方向 | 提问模板 | 适用场景 |
|---------|---------|---------|
| AI 智能 | "能否用AI自动完成XX？或提供智能推荐？" | 内容生成、智能分类、预测分析 |
| 实时能力 | "能否实时推送/同步XX信息？" | 协作编辑、消息通知、状态更新 |
| 自动化 | "能否自动触发XX，减少人工操作？" | 定时任务、事件驱动、工作流引擎 |
| 数据驱动 | "能否基于用户行为数据优化XX？" | 个性化推荐、A/B测试、智能排序 |

**执行方式：** 从上表中选取 2-3 个最相关的技术方向，用对应的提问模板向用户提问，确认用户感兴趣的技术创新点。

### 环节五：架构能力识别

基于 UniWeb + SaaS 技术栈，识别需求中已被架构覆盖的功能，标注"架构已支持"。

**目的**：避免对框架已提供的基础能力做重复需求设计，同时让用户了解平台自带能力。

**UniWeb + SaaS 架构已支持的能力清单**：

| 能力 | 来源 | 说明 |
|------|------|------|
| 用户认证（登录/登出/Token） | uw-auth-service | 多用户类型（ROOT/OPS/ADMIN/SAAS/MCH/GUEST/RPC）认证，支持oauth token技术方案 |
| 权限控制（RBAC） | uw-auth-service | @MscPermDeclare 注解，支持 USER/PERM/SUDO 等授权类型 |
| 多因素认证（MFA） | uw-mfa | IP限制、验证码、设备码、TOTP |
| SAAS 租户管理 | saas-base | SaasInfoHelper：租户信息查询、启用/停用 |
| 商户管理 | saas-base | SaasMchHelper：商户信息查询、保存、销量统计 |
| 系统消息通知 | saas-base | MsgHelper：短信、邮件发送 |
| 对象存储（OSS） | saas-base | SysOssHelper：上传参数生成、下载地址获取 |
| 数据字典 | saas-base | SysDictHelper / SaasDictHelper：系统/租户级字典 |
| 地区信息 | saas-base | SysAreaHelper：省市区数据 |
| 实时通知推送 | uw-notify-client | NotifyClientHelper：基于 SSE 的 Web 通知 |
| 分布式缓存 | uw-cache | FusionCache（本地+Redis）/ GlobalCache（纯Redis）/ GlobalLocker（分布式锁） |
| 异步任务调度 | uw-task | TaskCroner（定时任务）/ TaskRunner（异步队列） |
| 操作日志 | uw-auth-service | @MscPermDeclare log 级别：NONE/REQUEST/ALL/CRIT |
| 数据审计历史 | uw-common-app | SysDataHistoryHelper：数据变更记录 |
| 国际化（i18n） | uw-common-app | LocaleHelper：多语言支持 |
| 支付与计费 | saas-finance | 提供支付通道，在线支付，退款，余额管理、对账等财务相关功能 |
| 第三方接口接入 | saas-base (AIS) | Linker 机制：邮件、短信、支付等可插拔接入 |
| 应用功能与授权计费 | saas-base (AIP) | Vendor 机制: 功能授权、计费扣减 |
| 数据库分库分表 | uw-mydb-client | 自动分片、读写分离 |
| 日志收集 | uw-log-es / uw-logback-es | ES 日志存储与查询 |

**执行步骤：**

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 比对需求与上表 | 识别用户需求中已被架构覆盖的功能 |
| 2 | 列出已支持清单 | 向用户展示"以下功能架构已支持，无需额外设计" |
| 3 | 标注差异 | 如需超出架构默认能力（如自定义认证流程），标注为"需定制" |
| 4 | 确认 | 用户确认已支持清单和需定制项 |

**展示格式：**

向用户展示架构已支持能力时，使用以下格式：

**标题**：架构已支持能力（无需额外设计）

**表格**：
- 表头：功能 | 架构来源 | 默认支持 | 是否需要定制
- 行内容：根据实际识别出的能力填充

**示例行**：
- 用户登录/登出 | uw-auth-service | ✓ | -
- 角色权限管理 | uw-auth-service | ✓ | -
- 租户管理 | saas-base-common | ✓ | -
- 商户管理 | saas-base-common | ✓ | 需增加XX字段（如有定制需求）

**结尾说明**：以上功能由 UniWeb + SaaS 架构提供，在需求文档中标注"架构已支持"即可。

**注意事项：**

| 原则 | 说明 |
|------|------|
| 不省略展示 | 即使架构已支持，也要向用户展示这些能力，让用户了解平台能做什么 |
| 区分默认与定制 | 默认能力无需设计，但超出默认范围的定制仍需需求描述 |
| 不影响功能汇总 | 已支持功能仍出现在功能汇总表中，但分类标注为"架构已支持" |

### 环节六：功能汇总归类

将前5个环节产出的所有创意和功能，按 Must/Should/Could/架构已支持 初步归类。

**执行步骤：**

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 汇总所有功能 | 来自用户原始需求 + 场景扩展 + 竞品对标 + SCAMPER + 技术赋能 + 架构识别 |
| 2 | 去重合并 | 相同功能合并，标注来源 |
| 3 | 初步分类 | Must / Should / Could / **架构已支持** |
| 4 | 展示请用户筛选 | 用户可升级、降级或删除任何功能 |

**汇总表格式：**

向用户展示功能汇总时，使用以下结构：

**标题**：功能汇总（共 X 项）

**表格**：
- 表头：来源 | 功能 | 初步分类 | 说明
- 行内容：按实际识别出的功能填充

**示例行**：
- 用户原始需求 | XX管理 | Must | 核心业务路径
- 场景扩展 | 离线模式 | Should | 边缘场景但影响体验
- 竞品对标 | 数据导出 | Should | 行业标配
- SCAMPER | 一键登录 | Could | 提升体验但非必需
- 技术赋能 | AI智能推荐 | Could | 创新亮点
- 架构识别 | 用户认证/权限 | 架构已支持 | uw-auth-service
- 架构识别 | 租户/商户管理 | 架构已支持 | saas-base-common
- 架构识别 | 文件上传下载 | 架构已支持 | saas-base-common

**确认问题**：
- 有没有想要升级或降级的？
- 有没有不需要的可以去掉？
- 还有遗漏的吗？

确认后进入下一阶段，逐个深入澄清每个功能的细节。

### 头脑风暴阶段注意事项

| 原则 | 说明 |
|------|------|
| 数量优先于质量 | 先发散再收敛，不急于否定任何想法 |
| 标注来源 | 每个功能创意标注来源（用户/竞品/发散/技术），便于后续追溯 |
| 不做深度设计 | 本阶段只产出功能列表，不涉及交互细节和实现方案 |
| 用户主导筛选 | 智能体提供创意和分类建议，最终决策权在用户 |

## 1.3 产品架构规划（Architecture）

用户角色和终端平台是 N:N 关系，必须先确定角色，再为每个角色选择平台。角色表、平台表、命名规则和常见组合见 [role-platform-matrix.md](../../0-init/references/role-platform-matrix.md)。

### 执行方式（重要）

**逐个步骤确认，每步完成后再进入下一步**：

| 步骤 | 操作 |
|------|------|
| 1 | 使用 `AskUserQuestion` 呈现第一个确认问题（选择框） |
| 2 | **等待用户选择或自由录入** |
| 3 | 再确认下一个角色/平台（如有） |
| 4 | 每步都需用户确认后再继续 |

**禁止行为**：
- 一次性列出所有角色和平台组合让用户批量确认
- 纯文本列出选项让用户手工录入（必须用选择框）

### 第一步：确定用户角色（确认项 #1）

根据阶段1.1至1.2的讨论，逐个确认系统涉及的角色：
严格按照 [角色类型定义](../../0-init/references/role-platform-matrix.md#角色类型) 中的定义。

**执行方式**：
1. 先问："系统有哪几类用户？"
2. 用户回答后，**针对每个角色逐个确认**
3. 一个角色确认后再问下一个

**问题格式（使用 AskUserQuestion 选择框）**：

选项为通用参考，根据上下文动态调整。

| header | question | 参考选项（可动态调整） | multiSelect |
|--------|----------|----------------------|-------------|
| 用户角色 | 系统有哪几类用户？ | 普通用户/消费者、SAAS管理员、总后台管理员、SAAS商户 | true |

针对每个角色追问：
| header | question | 参考选项（可动态调整） | multiSelect |
|--------|----------|----------------------|-------------|
| 角色职责 | {role}的核心职责是什么？ | 使用核心功能、管理/审核、配置/设置、查看数据/报表 | true |

### 第二步：为每个角色选择平台（确认项 #2）

逐个角色确认使用的终端平台（一个角色可对应多个平台）：
严格按照 [终端平台类型定义](../../0-init/references/role-platform-matrix.md#终端平台) 中的定义。

**问题格式（使用 AskUserQuestion 选择框）**：

| header | question | 参考选项（可动态调整） | multiSelect |
|--------|----------|----------------------|-------------|
| 终端平台 | {role}主要在什么设备上使用？ | PC端网页、移动端H5、移动端APP/小程序、平板 | true |
| 多端支持 | 是否需要多端支持？ | 只需要一端、需要多端（请说明） | false |

> 提示：这里说"guest端"是口语化简称，正式命名使用"用户UniApp端"、"用户Web端"等"角色名+平台名+端"格式。

常见组合参考 [role-platform-matrix.md](../../0-init/references/role-platform-matrix.md#常见组合示例)。

### 第三步：确定项目清单（确认项 #3）

按 [命名规则](../../0-init/references/role-platform-matrix.md#项目命名规则) 汇总所有项目，请用户确认：

**示例确认话术：**

向用户展示项目清单时，使用以下结构：

**标题**：根据我们的讨论，项目清单如下

**后端项目**：
- nova

**前端项目**：
- nova-guest-uni（用户UniApp端）
- nova-guest-web（用户Web端）
- nova-mch-web（商户Web端）
- nova-admin-web（后台管理员Web端）

**确认问题**：
1. 角色和平台的组合是否正确？
2. 有没有遗漏？
3. 项目命名是否符合预期？

### 第四步：模块划分与优先级（确认项 #4）

每个前端项目独立拆分模块，确定功能优先级。**模块数量不设上限**，按业务实际需要划分，通常 5-20 个模块，复杂项目可达 30+。

> ⚠️ 不要人为压缩模块数量。如果一个功能域包含多个独立子功能，应拆分为独立模块，而非合并为一个"大模块"。

```
根据您的需求，建议功能优先级如下：
✅ Must have（P0）：列出核心功能（逐一列出，不要合并）
⏳ Should have（P1）：列出重要但可延后功能
📋 Could have（P2）：列出锦上添花的功能
🚫 Won't have：明确本期不做
```

**模块增删确认循环**（用户未确认前不得跳过）：

```
您同意以上模块划分吗？
- 如果需要新增：请告诉我模块名称和它要解决的核心问题
- 如果需要删减：请告诉我哪个模块可以去掉以及原因
- 如果需要调整优先级：请告诉我哪个模块需要升级或降级
```

| 用户反馈 | 追问策略 | 结束条件 |
|---------|---------|---------|
| 新增模块 | "这个模块的核心用户故事是什么？它与现有模块有重叠吗？" | 用户明确描述边界后重新展示完整清单 |
| 删减模块 | "去掉后{相关场景}怎么处理？是否影响{关联角色}？" | 用户确认替代方案或接受影响后重新展示清单 |
| 调整优先级 | "升级到P0意味着{其他模块}需要降级，可以接受吗？" | 用户确认优先级取舍后重新展示清单 |
| 无异议 | 复述最终清单请用户确认 | 用户说"确认"或"没问题" |

每次用户提出修改后，重新展示更新后的完整模块清单，再次询问"还有需要调整的吗？"直到用户明确确认无异议。

**模块依赖确认**（用户确认模块清单后追问）：

```
这些模块之间有先后依赖吗？比如必须先做完{模块A}才能做{模块B}？
```

| 依赖类型 | 示例 | 记录方式 |
|---------|------|---------|
| 数据依赖 | "订单"依赖"商品"数据 | 在模块文档中标注"依赖模块：{module}" |
| 流程依赖 | "审批"必须在"提交"之后 | 在流程图中体现先后顺序 |
| 无依赖 | 模块间完全独立 | 标注"无依赖" |

**模块间交互确认**（依赖确认后追问）：

```
这些模块之间需要互相调用吗？调用时传递什么数据？
```

| 交互类型 | 示例 | 记录方式 |
|---------|------|---------|
| 同步调用 | "下单"时调用"库存"检查余量 | 在模块文档中标注"调用：{module}→检查{数据}" |
| 异步通知 | "支付成功"后通知"订单"更新状态 | 在模块文档中标注"异步通知：{事件}→{目标模块}" |
| 数据传递 | "订单"传递订单ID（非完整对象）给"物流" | 标注传递方式：ID / 完整对象 / 事件 |
| 无交互 | 模块间无直接调用 | 标注"无交互" |

### 第五步：非功能需求确认（确认项 #5）

**问题格式（使用 AskUserQuestion 选择框）**：

| header | question | 参考选项（可动态调整） | multiSelect |
|--------|----------|----------------------|-------------|
| 用户规模 | 预期用户规模？ | < 1万、1-10万、10-100万、> 100万 | false |
| 性能要求 | 页面加载时间要求？ | < 1秒、1-3秒、3-5秒、无严格要求 | false |
| 安全要求 | 数据安全要求？ | 无特殊要求、基础加密、高安全（金融级） | false |
| 兼容性 | 需要支持的浏览器/设备？ | 仅现代浏览器、包含IE、移动端为主、全平台 | true |

## 1.4 站点地图与页面规划（Sitemap & Pages）

**仅对前端项目执行**，后端 app 项目不参与本阶段。

### 目标

将1.3确认的模块功能映射为用户可见的页面结构，确定完整的站点地图、页面层级和导航关系。

**核心认知**：
- 模块是功能的抽象（后端视角）—— 回答"系统做什么"
- 页面是功能的表象（用户视角）—— 回答"用户看到什么、怎么操作"
- 页面之间的交互直接决定用户体验，必须在PRD中以页面为核心组织

### 前置条件

阶段1.3（产品架构规划）已完成，模块划分和优先级已确认。

### 执行方式（重要）

**逐个前端项目执行，每个项目按5步推进，每步确认后再继续**：

| 步骤 | 确认项 |
|------|--------|
| 第一步 | 导航结构（Tab/侧边栏/浮动按钮等全局导航方式） |
| 第二步 | 站点地图（完整页面层级树） |
| 第三步 | 前端M→领域M映射 |
| 第四步 | 页面间跳转关系 |
| 第五步 | 用户确认完整的页面地图 |

### 第一步：确认导航结构

对每个前端项目，确认全局导航方式。使用 `AskUserQuestion` 逐项确认：

| 确认项 | 提问方向 |
|--------|---------|
| 底部Tab栏 | 几个Tab？每个Tab的名称和图标？ |
| 侧边栏/抽屉 | 是否有？从哪里触发？包含哪些入口？ |
| 浮动按钮 | 是否有全局浮动操作（如AI Copilot）？位置和行为？ |
| 顶部导航 | 页面内的子导航方式（Tab切换/筛选等）？ |

**向用户展示格式**（ASCII图）：

```
┌─────────────────────────────────────┐
│ [导航元素1] [导航元素2]    [其他按钮] │  顶栏
├──────────┬──────────┤               │
│  Tab1    │  Tab2    │               │  底部Tab
│          │          │               │
│          │          │               │
├──────────┴──────────┤               │
│     底部Tab栏        │               │
└─────────────────────────────────────┘
说明：全局浮动元素（位置、触发方式）
```

### 第二步：确认站点地图

逐个列出所有页面，形成页面层级树。**必须覆盖所有页面**：

| 页面类型 | 说明 |
|---------|------|
| Tab主页面 | 每个Tab的默认页面 |
| 子页面 | 从主页面跳转进入的页面 |
| 弹窗/浮层 | 非独立页面，在父页面上弹出 |
| 特殊状态页面 | 登录页、空状态、错误页等 |

**向用户展示格式**：

```
项目名: {project-name}-{role}-{platform}

Tab1: {Tab名称}
├── {P01-主页面名称}
├── {P02-子页面A}
└── {P03-子页面B}

Tab2: {Tab名称}
├── {P09-主页面名称}
└── ...

全局元素:
├── {D01-弹窗/浮层A}
└── {D02-弹窗/浮层B}

抽屉/侧边栏:
├── {P26-页面A}
└── {P27-页面B}
```

**编号规则**：
- M{seq} 一级菜单/功能组，序号从01递增
- M{seq}P{seq} 页面组（二级菜单），P序号从01递增

### 第三步：前端M→领域M映射

将1.3确认的领域模块与1.4第一步、第二步确认的前端M进行映射，向用户展示映射表：

| 前端M | 说明 | 对应领域 |
|-------|------|----------|
| {前端项目}/M01-{name} | {说明} | [M01-{name}](../../domains/M01-{name}.md) |
| {前端项目}/M02-{name} | {说明} | [M02-{name}](../../domains/M02-{name}.md) |

**规则**：多个前端M可能对应同一个领域M（如guest的"M01-社区"和saas的"M02-社区管理"都对应领域的"M06-社区与内容"）。

### 第四步：页面间跳转关系

描述核心用户流程中的页面跳转路径：

```
{起始页面} → {操作/触发条件} → {目标页面}
{起始页面} → {操作/触发条件} → {目标页面}
```

### 第五步：用户确认

汇总展示完整的站点地图（导航结构 + 页面清单 + 前端M→领域M映射 + 跳转关系），请用户确认。

> **门控规则**：站点地图未确认前，禁止进入1.5需求澄清阶段。

### 1.4 站点地图检查清单

| # | 检查项 | 通过条件 |
|---|--------|---------|
| 1 | 全局导航结构已确认 | Tab/侧边栏/浮动按钮已明确 |
| 2 | 所有页面已列出 | 含Tab主页、子页面、弹窗/浮层 |
| 3 | 前端M→领域M映射已完成 | 每个领域M至少对应1个前端M，每个前端M对应1个领域M |
| 4 | 页面间跳转关系已描述 | 核心流程的跳转路径清晰 |
| 5 | 用户已确认完整站点地图 | 用户说"确认"或"没问题" |

## 1.5 需求澄清与验证（Page-Based Clarification）

从按模块澄清改为**按页面澄清**。每个页面的澄清自动覆盖其包含的所有模块功能，确保前端开发者拿到每个页面的完整规格。

### 执行规则

| 规则 | 说明 |
|------|------|
| 执行顺序 | P0前端项目 → P1前端项目 |
| 执行粒度 | 一次聚焦一个项目，按页面逐一澄清 |
| 澄清单位 | 每个页面独立完成7步澄清 |
| 同角色复用 | 同一角色的用户故事仅澄清一次，后续项目复用 |
| 共享模块 | 跨项目共有模块仅澄清一次（在首次出现的页面中完成） |

### 单页面澄清7步法

对每个页面，按以下7步逐一执行。**每步一问，等待用户回答后再进入下一步**。

| 步骤 | 维度 | 目标 | 提问示例 |
|------|------|------|---------|
| 1 | 页面定位 | 确认页面类型、所属导航位置、核心使命 | "这个页面的核心目的是什么？用户来这里要完成什么事？" |
| 2 | 页面元素 | 确认页面包含的所有UI区域和数据展示 | "页面上需要展示哪些信息？分成几个区域？" |
| 3 | 交互流程 | 确认用户在该页面可执行的所有操作及系统响应 | "用户在这个页面上可以做什么操作？每个操作后系统如何响应？" |
| 4 | 异常路径 | 确认操作失败、网络超时、数据冲突等异常场景的处理方式 | "如果这个操作失败了（如网络超时、数据冲突），用户会看到什么？可以怎么处理？" |
| 5 | 状态设计 | 确认页面的不同状态（空/加载/正常/异常/业务状态） | "这个页面在无数据时展示什么？出错时怎么办？有哪些特殊状态？" |
| 6 | 线框图确认 | 用ASCII绘制页面线框图，向用户确认布局 | "基于以上讨论，我画了这个页面的线框图，请确认布局是否符合预期？" |
| 7 | 复述验证 | 用自己的话复述页面规格，请用户确认 | "让我复述一下这个页面：{复述}，是否准确？" |

**步骤1提问模板**：

```
AskUserQuestion({
  questions: [{
    question: "关于M{seq}P{seq}-{page}，这个页面组的核心目的是什么？",
    header: "页面定位",
    options: [
      { label: "浏览查看", description: "用户来这个页面主要查看信息" },
      { label: "操作执行", description: "用户来这个页面完成某个操作" },
      { label: "信息录入", description: "用户来这个页面填写/提交信息" },
      { label: "综合功能", description: "以上多种目的的组合" }
    ],
    multiSelect: false
  }]
})
```

**步骤5线框图绘制原则**：

| 原则 | 说明 |
|------|------|
| 必须绘制 | 每个页面必须有ASCII线框图 |
| 真实比例 | 反映移动端（48-60字符宽）或桌面端（80+字符宽）布局 |
| 标注区域 | 每个功能区域用分隔线区分，右侧标注区域名称 |
| 包含数据 | 用真实示例数据填充，不用"XX"等占位符 |
| 动态部分 | 如有Tab/状态切换，可画多个线框图对比 |

### 页面级检查清单

每个页面完成7步后，对照以下清单确认：

| # | 检查项 | 通过条件 |
|---|--------|---------|
| 1 | 页面定位已明确 | 类型、所属导航、核心使命已确认 |
| 2 | 所有区域已列出 | 无遗漏的UI区域 |
| 3 | 所有交互已描述 | 每个可点击/可操作的元素都有响应说明 |
| 4 | 异常路径已覆盖 | 操作失败、超时、冲突等异常场景有处理方案 |
| 5 | 所有状态已覆盖 | 空状态、加载中、正常、异常、业务状态 |
| 6 | 线框图已确认 | 用户确认布局符合预期 |
| 7 | 复述已通过 | 用户确认复述准确 |

**任一项未通过，需补充澄清后重新验证。**

### 项目级页面澄清流程

完成一个项目的所有页面后，进行项目级检查：

| 检查项 | 说明 |
|--------|------|
| 页面完整性 | 站点地图中的每个页面都已完成澄清 |
| 跳转一致性 | A页面说跳转到B，B页面的入口确实存在 |
| 状态连续性 | 跨页面的状态流转无断裂 |
| 数据来源 | 每个页面展示的数据都有对应的数据来源说明 |

## 1.6 汇总确认（Final Confirmation）

### 执行方式（重要）

**逐个问题确认，一问一答**：

| 步骤 | 操作 |

| 1 | 展示汇总文档 |

| 3 | **等待用户回答** |

**禁止行为**：一次性列出3个问题让用户逐一回答。

## 需求访谈汇总

### 站点地图（前端项目）

|------|-------------|---------|-------------|

### 最终确认三问（使用 AskUserQuestion 选择框）

向用户逐个提出以下问题，全部确认后才允许进入文档编写。

| # | header | question | 参考选项（可动态调整） | multiSelect |
|---|--------|----------|----------------------|-------------|

以上是我们经过多轮讨论确认的所有需求信息。

| 用户提出补充或调整 | 🔄 回到对应阶段补充后重新汇总 |
| 用户对功能范围有异议 | 🔄 回到1.3重新划分 |

## 流程裁剪规则

根据项目复杂度自动简化访谈流程，避免简单项目过度访谈：

| 条件 | 裁剪方式 |
|------|---------|
| 仅1个前端项目 + ≤5个页面 | 1.5可连续完成，不强制拆分项目切换 |
| 无可参照竞品 | 1.2环节二（竞品功能映射）标记"无竞品参考"并跳过 |
| 内部工具/后台系统 | 1.2环节三（SCAMPER）和环节四（技术赋能）可选跳过 |
| 用户表示"赶时间" | 每个页面的复述验证可简化为关键页面验证（核心流程页面必验证，辅助页面可抽样） |
| 用户主动要求跳过某阶段 | 允许跳过，在访谈记录中标注"用户主动跳过"及原因，后续发现遗漏时不回溯 |

## 启发式提问工具箱

### 5 层 Why 方法

当用户提出一个需求时，连续追问5层"为什么"挖掘真实痛点：

```
用户："我需要一个报表功能"
  Why 1："为什么需要报表？"
    → "要看销售数据"
  Why 2："为什么要看销售数据？"
    → "了解哪些产品卖得好"
  Why 3："了解之后呢？"
    → "调整采购策略"
  Why 4："现在为什么调整不了？"
    → "因为数据散落在各个Excel里，汇总要2天"
  Why 5："如果汇总只要10分钟呢？"
    → "那就能每周调整一次采购，而不是现在每月一次"

真实需求：自动化数据汇总，而非"报表功能"
```

### 苏格拉底式提问库

| 类型 | 问题模板 | 使用场景 |
|------|---------|---------|
| 澄清型 | "您说的XX，具体是指什么？能举个例子吗？" | 用户描述模糊时 |
| 假设型 | "如果用户操作错了，您希望系统怎么反应？" | 用户只考虑正常流程时 |
| 证据型 | "您是怎么知道用户需要这个的？有数据支撑吗？" | 用户凭感觉提需求时 |
| 视角型 | "从XX角色的角度看，这个功能意味着什么？" | 多角色场景 |
| 后果型 | "如果这个功能做不了，最坏的情况是什么？" | 验证功能必要性 |
| 反向型 | "如果这个功能消失了，用户会怎么抱怨？" | 验证核心价值 |
| 类比型 | "这个功能像现实生活中的什么？" | 帮助用户表达抽象需求 |

### 反向验证技巧

| 技巧 | 话术 | 目的 |
|------|------|------|
| 功能消失测试 | "如果明天这个功能突然没了，用户最直接的替代方案是什么？" | 区分 Must have 和 Should have |
| 极简版本测试 | "如果只能用文字描述，您会怎么向用户解释这个功能？" | 剥离装饰，找到核心价值 |
| 竞品差评法 | "竞品做了类似功能，你觉得他们哪里做得不好？" | 找到差异化机会 |
| 极端场景测试 | "如果用户量突然翻10倍，这个功能还能用吗？" | 发现潜在瓶颈 |

## 示例对话

### 示例1：B2B销售工具

**用户**：我想做一个任务管理APP。

**助手**：好的，市面上有很多任务管理工具，**为什么你觉得还需要做一个新的？** 是想解决现有工具解决不了的特定问题，还是有特殊的用户群体？

**用户**：我们公司销售团队需要管理客户跟进任务。

**助手**：明白了，B2B销售场景。让我深入了解一下：**一个销售代表典型的一天是怎样的？** 比如早上来公司第一件事做什么？在什么场景下会创建任务？目前他们用什么工具记录这些信息？

**用户**：他们用Excel表格记录客户和跟进日期。

**助手**：好的，从Excel迁移到APP。**当他们打开Excel时，最常遇到的3个痛点是什么？** 是提醒不及时？数据不同步？还是查找历史记录困难？

**用户**：主要是提醒不及时，经常忘了跟进。还有就是经理看不到团队的跟进情况。

**助手**：好的，让我确认我的理解：
- 核心用户：一线销售代表（25-35岁，高频外出，手机操作为主）
- 核心痛点：跟进提醒不及时 + 经理无法监控团队进度
- 现有方案：Excel手动记录

**这个理解正确吗？接下来我想深入了解具体的功能需求。**

**用户**：对，就是这样。

**助手**：好的。那我们来聊第一个核心功能。**当销售代表打开APP，你希望TA看到的第一个界面是什么？** 是今日待办列表？还是客户列表？为什么？

[... 继续深入 ...]

### 示例2：C端电商

**用户**：做一个生鲜电商小程序。

**助手**：好的。**为什么选择生鲜这个品类？** 是有供应链优势，还是看到了特定的市场机会？

**用户**：我们有自己的农场，想直接卖给消费者。

**助手**：理解了，从农场直达消费者。**目前的销售渠道是什么？** 线下集市？社区团购？还是纯线上？

**用户**：主要是社区团购，团长在微信群下单。

**助手**：那现在团长在微信群里管理订单，**最麻烦的环节是什么？** 是收款对账？统计订单？还是售后处理？

**用户**：每天晚上要统计几十个群的订单，很容易出错。

**助手**：让我确认一下：核心问题是**团长手工统计订单效率低且容易出错**。那这个小程序是给**团长用的**，还是给**消费者直接下单的**？

**用户**：都有。消费者在群里点链接就能下单，团长后台能看到所有订单。

**助手**：明白了，两端都需要。那我们来确认一下：
- **用户角色**：guest（消费者）+ mch（团长）+ admin（后台管理）
- **终端平台**：guest用微信小程序下单，mch用Web管理订单，admin用Web管理全局

对应的项目清单：
- `nova`（后端）
- `nova-guest-uni`（用户微信小程序端）
- `nova-mch-web`（商户Web端）
- `nova-admin-web`（后台管理员Web端）

**这个划分准确吗？先从哪个角色开始深入？**

## 追问策略

### 模糊回答 → 具体化

| 用户回答 | 追问话术 |
|---------|---------|
| "要快" | "具体来说，页面加载在几秒内可以接受？" |
| "要好用" | "能描述一下您觉得'好用'的具体体验吗？比如一键完成某个操作？" |
| "参考XX" | "跟XX相比，您希望在哪些方面做得更好或不同？" |
| "看情况" | "能举一个最常见的具体情况吗？我们先把最典型的场景确定下来。" |
| "都要" | "如果必须排个优先级，最先做哪个？" |
| "和XX一样就行" | "XX有那么多功能，您最常用的是哪几个？" |

### 技术术语 → 转化为用户语言

| 避免说 | 应该说 |
|--------|--------|
| "需要RBAC权限模型吗？" | "不同的人登录后看到的内容和操作权限需要不同吗？" |
| "需要分布式缓存吗？" | "预计同时使用的人最多有多少？" |
| "需要消息队列吗？" | "有些操作需要后台慢慢处理，还是必须立刻完成？" |
| "需要微服务架构吗？" | "未来有哪些功能模块可能独立扩展？" |

### 沉默或不确定 → 提供选项

当用户对某个问题不确定时，提供2-4个选项供选择：

```
这个问题您可能还没想清楚，我给您几个常见的选择：

A. {选项1} - {说明}
B. {选项2} - {说明}
C. {选项3} - {说明}
D. 暂时跳过，后续再定

您倾向哪个？
```

### 建设性挑战

当用户提出可能不合理的需求时，温和质疑：

| 场景 | 话术 |
|------|------|
| 功能过多 | "这些功能都很好，但如果2周内必须上线，您觉得哪些可以第一批做？" |
| 过度设计 | "这个功能确实能提升体验，但实现成本较高。先做个简化版本可以吗？" |
| 自相矛盾 | "您之前说目标是简单易用，但这个功能可能增加操作步骤。您觉得怎么平衡？" |
